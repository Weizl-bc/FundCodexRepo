# 基金基础数据与持仓管理 表接口设计

## 1. 设计范围

基于以下两个 PRD 设计：

- `prd/数据中心/基金基础数据.md`
- `prd/我的基金/持仓管理.md`

本设计覆盖：

- 基金基础数据列表
- 基金详情
- 持仓管理列表
- 持仓详情总览
- 基金走势图
- 基金重仓股
- 行业分布

## 2. 设计原则

1. 基金主数据与用户持仓数据分层，避免耦合。
2. 行情快照、重仓股、行业分布按“基金 + 日期”存储，便于去重和历史追溯。
3. 持仓以“当前持仓快照”展示为主，同时保留交易流水扩展位。
4. 接口优先满足前端页面展示，不强行过度抽象。
5. 由于 PRD 中明确提到“手动录入持仓时自动刷基金数据入库，且需要去重”，因此所有外部抓取表都提供唯一约束。

  - `total_cost_amount`：持仓成本金额
- 页面展示时：
  - “持有金额”取 `current_amount`
  - “持仓金额”取 `total_cost_amount`

## 4. 推荐表关系

- `fund_info.fund_code` -> 基金主键业务编码
- `fund_nav_snapshot.fund_code` -> `fund_info.fund_code`
- `fund_nav_history.fund_code` -> `fund_info.fund_code`
- `fund_top_holding.fund_code` -> `fund_info.fund_code`
- `fund_industry_distribution.fund_code` -> `fund_info.fund_code`
- `user_fund_position.fund_code` -> `fund_info.fund_code`
- `user_fund_trade.fund_code` -> `fund_info.fund_code`

说明：

- 如果后端倾向于纯性能导向，可以直接用 `fund_code` 做关联。
- 如果后续需要做主数据治理，再补 `fund_id` 也可以，不影响当前页面开发。

## 5. 页面接口设计

接口前缀建议统一为 `/api`.

### 5.1 基金基础数据列表

`GET /api/funds`

查询参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| keyword | string | 否 | 基金名称或代码模糊搜索 |
| fundCode | string | 否 | 精确基金代码 |
| fundName | string | 否 | 基金名称模糊搜索 |
| pageNum | int | 否 | 默认 1 |
| pageSize | int | 否 | 默认 20 |
| sortBy | string | 否 | `dailyGrowthRate` |
| sortOrder | string | 否 | `asc` / `desc` |

返回示例：

```json
{
  "pageNum": 1,
  "pageSize": 20,
  "total": 2,
  "list": [
    {
      "fundCode": "161725",
      "fundName": "招商中证白酒指数",
      "dailyGrowthRate": 1.23,
      "tradeDate": "2026-03-11"
    }
  ]
}
```

### 5.8 手动录入持仓

`POST /api/positions`

请求示例：

```json
{
  "userId": 1,
  "fundCode": "161725",
  "fundName": "招商中证白酒指数",
  "totalAmount": 8000.1234,
  "avgCostNav": 1.3556,
  "totalCostAmount": 10800.00,
  "firstBuyDate": "2025-06-15",
  "lastTradeDate": "2026-03-01"
}
```

处理逻辑建议：

1. 先写入或更新 `user_fund_position`
2. 若 `fund_info` 不存在，则自动抓取并写入基金主数据
3. 同步刷入最近基金快照、走势图、重仓股、行业分布
4. 对快照类数据按唯一键去重

返回示例：

```json
{
  "success": true,
  "message": "持仓录入成功，基金基础数据已同步"
}
```

## 7. 实现建议

1. 先实现只读接口：
   - `/api/funds`
   - `/api/funds/{fundCode}`
   - `/api/funds/{fundCode}/trend`
   - `/api/funds/{fundCode}/top-holdings`
   - `/api/funds/{fundCode}/industry-distributions`
   - `/api/positions`
   - `/api/positions/{fundCode}`

2. 再实现持仓录入接口：
   - `/api/positions`

3. 持仓录入后触发基金数据同步任务：
   - 若基金不存在则创建
   - 若当日快照不存在则插入
   - 历史走势按日期去重
   - 重仓股和行业分布按报告期去重覆盖

4. 计算规则建议：
   - `current_amount = total_amount * latest_nav`
   - `holding_profit = current_amount - total_cost_amount`
   - `holding_profit_rate = holding_profit / total_cost_amount * 100`
   - `holding_days = 今天 - first_buy_date`
   - `position_ratio = current_amount / 用户总持仓金额 * 100`

## 8. 当前设计中的假设

1. 当前系统支持多用户，因此持仓表带 `user_id`。
2. “当日走势”在列表页中按“当日涨跌幅”展示。
3. 详情页中的走势图、重仓股、行业分布，在基金基础数据和持仓管理两个模块中复用同一套接口。
4. PRD 未要求交易流水页，但为了保证持仓数据未来可追溯，预留 `user_fund_trade`。
